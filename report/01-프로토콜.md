# 01 — 실험 프로토콜

## 1. 실험 여정 (단계 개요)

이 작업은 단일 라운드가 아니라 측정 패러다임이 바뀔 때마다 클린 슬레이트로 재시작한 여러 라운드의 연속이다.

| 단계 | 기간/위치 | 목적 | 산출 |
|---|---|---|---|
| **A100 초기** | `legacy/` | 최초 throughput·정확도 탐색 | A100은 **FP8 텐서코어 HW 가속 없음** 확인 → H100 필요 판단 |
| **H100 라운드1** | `legacy/h100-round1/` | 모델×정밀도×기법 전조합 throughput + ops 근본원인 태스크(N=30) | MTP 2.6×(dense)·양자화 무손실·diffusion 최速 1차 확인. 단 정확도는 합성 ops 태스크(천장 포화) |
| **Phase 1** | `legacy/`로 이관 | 정정 누적으로 비대해진 1차 산출물을 감사추적 보존 + 새 작업 분리 | `context/`(P2)·`evals/`(P3) 신설 |
| **Phase 2** | `context/` | "8K는 측정 선택이지 한계 아님" — 실효 컨텍스트 규명 | **전 셀 네이티브 256K 도달** + NIAH 검증, `max_context.csv` |
| **Phase 3** | `evals/`, `results_consolidated.csv` | 표준 벤치 배터리(3프레임워크) × thinking-on 동등조건 | 본 보고서 결과(03) |

핵심 전환점:
- 라운드1의 정확도는 **자체 합성 ops 태스크**(근본원인 서비스 선택, N=30)였고 Gemma가 전부 100%로 **천장 포화**되어 변별력이 없었다. → Phase 3에서 **표준 벤치(GPQA/MMLU-Pro/IFEval/한국어)**로 전환.
- 라운드1은 컨텍스트를 8K로만 쟀는데 이는 프롬프트 크기 선택이었다. → Phase 2에서 256K까지 규명.
- 라운드1은 thinking 일관성이 통제 안 됐다. → Phase 3에서 전 모델 thinking-on 동등조건으로 통일.

## 2. 동등조건 원칙 (Phase 3 본체)

**공정 비교 = 모든 모델에 동일한 관대한 멍석.** 토큰 예산으로 추론을 강제 종료(클램프)하는 것은 능력을 인위로 억제해 불공정하므로 금지한다. 동등은 "절단"이 아니라 "충분히 큰 동일 예산 + 동일 샘플링 + 전 모델 thinking-on"으로 달성한다.

| 조건 | 값 | 근거 |
|---|---|---|
| thinking | 전 모델 enable_thinking=true | Gemma4·Qwen3.6 둘 다 지원 → 켜야 공정 |
| 답 분리 | vLLM `--reasoning-parser gemma4\|qwen3` | thinking을 content에서 제거, 깨끗한 답만 채점 |
| 생성 길이 | max_gen_toks/max_tokens = 32768 | 인위 절단 0. finish_reason=length 비율 ≈0 확인(예: 31B 50/50 stop) |
| 샘플링 | temp 0.6 / top_p 0.95 | 전 모델·전 프레임워크 통일 |
| 프롬프트 | 동일 프롬프트·few-shot | 프레임워크 기본 |

effort(모델이 자연히 쓰는 추론 토큰량 — 같은 enable_thinking이라도 Gemma 3444 vs Qwen 7639처럼 모델마다 다름)는 **모델 특성으로 보고하되 인위로 맞추지 않는다**.

### 2.1 thinking-on 배선 (프록시)

평가 프레임워크(lm-eval/inspect/HRET)는 요청에 `chat_template_kwargs`를 직접 넣지 못한다 → thinking을 켤 방법이 없다. 이를 **프록시(`inject_proxy.py`, :8001)**가 모든 `/chat/completions`에 `enable_thinking` 주입으로 해결한다.

```
lm-eval / inspect / HRET  →  :8001 (inject_proxy, ENABLE_THINK=true)  →  :8000 (vLLM, --reasoning-parser)
```

세 프레임워크 전부 :8001을 경유하므로 동일한 thinking-on 조건이 보장된다.

## 3. 서빙 구성 (vLLM)

| 항목 | 설정 | 비고 |
|---|---|---|
| 이미지 | `vllm/vllm-openai:nightly`(Gemma/Qwen 네이티브), `:v0.22.1`(Qwen27, `--max-num-seqs 256`), `:gemma`(diffusion) | Gemma4는 nightly만 네이티브; stable은 폴백 후 크래시 |
| 정밀도 | bf16 / fp8(runtime) / int8(per-channel weight-only) / qat-w4a16(compressed-tensors) | int8은 **MoE/diffusion만 실효**, dense는 no-op |
| KV 캐시 | 무거운 dense bf16은 `--kv-cache-dtype fp8` 필수 | 31B-bf16은 auto-KV로 94K cap → kv-fp8로 256K |
| util | AR 0.92, MTP 0.85(드래프터 여유), Qwen35 bf16/int8은 0.97+eager | `gpu_memory_utilization`은 점유 상한 — **낮추면 KV가 작아짐**(직관 반대) |
| 기타 | `--no-enable-prefix-caching`, `VLLM_MEMORY_PROFILER_ESTIMATE_CUDAGRAPHS=0` | 측정 공정성 |

**MTP 드래프터:** Gemma는 **별도 ~0.4B 드래프터**(`gemma-4-<size>-it-assistant`, 31B-qat는 `...-qat-q4_0-unquantized-assistant`). Qwen은 **임베디드 MTP 헤드**(`--speculative-config '{"method":"mtp","num_speculative_tokens":2}'`).

**아키텍처 메모(왜 256K가 들어가나):** Gemma4 = sliding_attention×3 + full_attention×1(`sliding_window=1024`). Qwen3.6 = 두 모델 다 gated-delta 선형어텐션 하이브리드(linear×3 + full×1, full만 GQA). 두 계열 모두 토큰당 KV가 작아 256K가 여유롭게 들어감. (상세: `context/FINDINGS.md`)

## 4. 프레임워크 · 벤치마크

세 프레임워크는 **서로 다른 면**을 본다 — 이것이 "다면 평가"의 실체다.

| 프레임워크 | 벤치마크 | 보는 면 | 채점 |
|---|---|---|---|
| **lm-eval** 0.4.12 | GPQA-Diamond(full 198) | 영어 복합추론 | flexible-extract on content |
| **inspect_ai** + inspect_evals | GPQA-Diamond · MMLU-Pro · IFEval | 추론 + 지시준수 | `--epochs 1`, metric `final_acc` |
| **HRET** (haerae-eval-toolkit) | HAE-RAE Bench · KMMLU | 한국어 | 자체러너 마지막 (X) 추출 |

**샘플링 결정:** GPQA-Diamond(198)는 **full**. 초대형(MMLU-Pro 12K, KMMLU 35K)은 thinking-on이면 셀당 수 시간이라 N=500 샘플(사용자 승인). haerae=언어이해(전부 높음), kmmlu=전문지식(변별력 있음).

## 5. 측정 방법론

### 5.1 속도 (`bench_tp.py`)
- **total throughput = completion_tokens / 전체요청시간**(non-streaming). `ignore_eos=True`로 고정 512토큰 출력, temp=0, warmup 1회, prefix-cache off. short(짧은 프롬프트) + 8K(≈8K 입력) 2조건.
- **streaming `decode_tok/s` 금지**: spec decode는 토큰을 버스트로 커밋해 순간 토큰율이 왜곡 → MTP를 ~3배 과소측정(라운드1의 "MTP 손해" 오결론 원인, 02 참조).

### 5.2 컨텍스트 (`niah.py`, NIAH)
- 긴 로그(haystack) 깊이 10/50/90%에 needle(접근코드)을 심고 회수율 측정. 길이는 vLLM `/tokenize`로 정확히 보정(추정 아님). "로드 ≠ 작동"을 분리 검증.

### 5.3 MTP 무손실 검증 (이중 설계)
- **정확도**: MTP-on으로 GPQA full 198을 AR과 동일조건 직접 측정 → 셀별 Δ 비교(n=30 A/B는 underpowered라 정식 근거로 불채택).
- **토큰동일성**: greedy(temp=0) MTP-off vs on 토큰열 byte 동일성 — **반드시 대조군(같은 AR 두 번: off vs off2)과 함께** 해석(vLLM은 temp=0이라도 run-to-run 비결정일 수 있어, 대조군 없이는 분기 원인을 귀속 못 함).
