# 02 — 시행착오와 교훈

원칙: **no-hiding.** 깨진 측정·오진·정정을 숨기지 않고 "증상 → 오진 → 진짜 원인 → 해법 → 교훈"으로 기록한다. 잘못이 가장 큰 교훈을 낳았다.

---

## A. 측정 패러다임을 뒤집은 치명적 결함 (라운드1)

### A1. [치명적] streaming 측정이 MTP를 3배 과소측정
- **증상:** acceptance 100%인데 streaming으론 MTP 가속이 안 보임. Qwen에선 "0.44× 손해" 결론까지 나옴.
- **오진:** "MoE는 MTP에 불리", "검증 비용이 이득을 잡아먹는다".
- **진짜 원인:** streaming `decode_tok/s`는 spec decode의 버스트 커밋을 과소측정. vLLM "Mean acceptance length"로 교차검증하니 실제론 이득.
- **해법:** non-streaming 총처리량(고정출력/wall-clock)으로 전면 재측정 → "손해"가 **2.6~3.0× 이득**으로 반전.
- **교훈:** spec decode/MTP는 절대 streaming 토큰율로 재지 말 것. 단일 측정 결함이 Gemma·Qwen 양쪽 초기 결론을 통째로 무효화했다.

### A2. 유령 systemd 서비스가 GPU를 훔침
- **증상:** "H100에서 31B/fp16이 메모리 부족." (사용자: "A100에선 잘 돌았는데?")
- **진짜 원인:** A100 스냅샷에 `ops-llm-test.service`가 따라와 부팅 시 호스트 vLLM 자동 기동, GPU ~33GB 상시 점유.
- **해법:** `systemctl disable` + util 0.95. (31B는 H100에서 정상 — 크기 문제 아님.)
- **교훈:** 스냅샷 VM은 상속된 systemd 유닛을 의심. `nvidia-smi`로 "내가 안 띄운 프로세스"부터 확인.

## B. 답 추출 함정 — thinking-on이면 채점이 1순위 리스크 (Phase 3)

thinking을 켜면 채점기가 "추론 요약부의 첫 옵션"을 최종답으로 오집해 0.0·공백·random미만이 나온다. 전부 **점수가 깨진 뒤 풀런 낭비**로 발견했고, 이후 "limit 5 스모크 선행"의 근거가 됐다.

| 프레임워크/벤치 | 증상 | 진짜 원인 | 해법 |
|---|---|---|---|
| lm-eval GPQA | accuracy 0.0 | strict-match를 점수로 집음 | flexible-extract 우선(`ex_lmeval.py`) |
| lm-eval MMLU-Pro(thinking) | content 공백 → 0 | reasoning-parser가 thinking과 **답까지** 제거 | MMLU-Pro는 **inspect로** 이관 |
| inspect IFEval | 결과 공백 | dep 누락 + metric명이 `final_acc`(≠accuracy) | git dep 설치 + grep `(final_acc\|accuracy)` |
| inspect 전반 | n이 2~3배 | epochs 기본>1 | **`--epochs 1` 필수** |
| HRET haerae/kmmlu | random 미만(0.146) | string_match가 thinking 요약부 첫 옵션 오집 | 자체러너 + **마지막 (X)** 추출(`run_hret.py`, 0.146→0.844) |
| HRET 로딩 | DatasetNotFoundError | `cls(dataset_name=...)`로 기본 HF경로 덮어씀 | `load_datasets(ds_name)` (덮어쓰지 말 것) |

**교훈:** 채점기는 "추론 요약부의 첫 옵션"이 아니라 "최종 답"을 잡아야 한다. 보편 해법 = content의 **마지막** answer-letter / 마지막 `(X)`.

라운드1에도 같은 부류가 있었다 — Qwen 정확도 6.7%(2/30)는 실 성능이 아니라 `bench_acc.py` 추출기가 Qwen의 추론 서두를 서비스명으로 오집한 **파서 아티팩트**였다(Gemma는 포맷 지켜 100%). Phase 3의 강건 추출이 이를 정면 해결했다.

## C. vLLM 서빙 함정

### C1. KV OOM과 util 방향 착각
- 31B-bf16(62GB)은 auto-KV로 ~94K에서 OOM. **util을 0.88로 낮췄더니 더 악화**(KV가 더 작아짐) — `gpu_memory_utilization`은 점유 상한이라 **올려야** KV가 커진다. 해법: `--kv-cache-dtype fp8`로 KV 절반 → 256K. NIAH로 fp8 KV가 장거리 회수를 열화 안 시킴 확인.

### C2. eager 폴백 = 속도 붕괴 (정밀도 선택의 숨은 축)
- qwen35 bf16/int8(70GB)은 메모리가 빠듯해 CUDA graph 캡처 버퍼가 안 남아 **enforce-eager로 폴백 → 15 tok/s**(fp8 217의 1/14). 같은 모델·무손실 정확도인데 실행모드 하나로 14× 차이.
- **이 15 tok/s는 공정한 속도 측정치가 아니라 단일 H100 폴백값.** 정확도(0.808/0.788)만 유효 비교 대상. 메모리 여유 환경(더 큰 GPU·멀티GPU TP)에선 graph 적재로 정상화.
- 라운드1의 31B-bf16-MTP도 동일 — cudagraph OOM vs eager 왜곡으로 **단일 H100에서 깨끗한 측정 불가**(fp8/qat에선 정상이라 실무상 무의미한 코너로 NA 처리).
- **교훈:** "정확도 무손실"이라도 **graph 적재 가능 여부가 비용을 가른다.** 정밀도/배치/컨텍스트를 graph가 올라가는 선에서 잡는 게 실전 레버.

### C3. 디스크 churn / 재다운로드 충돌
- 50~67GB base 모델을 지웠다 받기 반복(라운드1: 부트디스크 194→500GB 리사이즈로 해결). Phase 3에선 R3 후 지운 diffusiongemma를 NIAH 때 재서빙하자 **재다운로드가 디스크풀(100%)을 유발해 serve 사망** → 불필요한 qwen35 base(67G) 제거로 공간 확보 후 재개.
- **교훈:** 모델 캐시 회전은 "나중에 다시 쓸 모델"을 확인하고. 디스크부터 확보.

### C4. Qwen int8 속도이득 0
- Qwen27 int8은 속도이득이 없다 — 선형/하이브리드 어텐션 recurrence 병목이지 가중치 대역폭 병목이 아니기 때문. int8은 dense엔 no-op, Qwen하이브리드엔 무이득. **측정해서 확인**해야 하는 사례.

### C5. sshd wedge / 오케스트레이션
- 부하 중(GPU+CPU 포화, 부하 중 kill, 무거운 foreground import) sshd가 wedge → SSH 255. 해법: `gcloud compute instances reset`(디스크 영속). 그래서 평가는 항상 **detached(setsid)·파일로깅**, 폴링은 가벼운 sudo만.
- pkill을 측정과 같은 SSH 채널에 섞으면 자기 부모 세션까지 끊김 → 분리 실행.
- 라운드1엔 **pgrep self-match 데드락**도 있었다 — waiter 가드 `pgrep -f run_rem.sh`가 자기 cmdline을 매치해 영영 실행 안 함. 해법: `flock`(커널 inode 락). **교훈: 프로세스 존재검사를 모니터 자신의 이름으로 하지 말 것.**

## D. 프로세스 규율 (잦은 재작업을 막는 5계명)

사용자의 "왜 미리 발견 못 했나"라는 질문에 대한 답 — 반복 수정의 근본 원인과 규율:

1. **풀런 전 `--limit 5` 스모크.** 각 벤치 1셀씩 limit 5로 점수 sane(>random, 공백 아님) 확인 후에만 풀런. 추출 버그는 수 시간 풀런 후가 아니라 5문항에서 잡는다. ← 가장 큰 교훈.
2. **실행 중인 bash 스크립트 절대 수정 금지.** bash는 파일을 증분으로 읽어, 도는 중 덮어쓰면 깨진 라인 실행("line N: command not found")으로 캠페인 사망. 새 파일로 빌드 후 재기동.
3. **추정 말고 측정.** "이건 인프라 한계라 안 됨" 단정 금지 — qwen35-bf16가 안 들어갈 거라 단정했으나 실측하니 fit(단 eager 15tok/s). 한 줄 probe로 확인 후 결론.
4. **재개 가능 설계.** 셀·벤치별 `.done` 마커 skip, 셀별 즉시 CSV 기록. Spot 선점/리셋에도 진행 보존.
5. **결론 과대주장 금지.** n=30 A/B는 CI ±0.14로 underpowered → "무손실 확정" 불가, full N으로 확정. MTP "토큰 무손실"도 토큰동일성+대조군 검증으로 **"비트-무손실"은 반증**하고 "정확도 무손실"만 확정(03-C). 사용자의 "무손실 결론은 위험" 지적이 이 규율을 낳았다.
